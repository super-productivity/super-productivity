## 1. The #9118 collision — do this first

**Why first:** it is the only item that can ship a silent data regression, and it
gets harder to see the longer the branch sits.

PR #9118 (`c18055a341`, fix for #9098 — "widget labels a stale list as Today")
merged to master **after** this branch was last rebased. It is currently the
master tip.

The two changes overlap badly:

- **#9118** added a `{ dayStr, validUntil }` stamp to
  `src/app/features/android/android-widget.model.ts` + `android-widget.selectors.ts`,
  and taught the Kotlin reader (`WidgetData.kt`) to derive the header from it via
  `WidgetData.headerFor` → `sealed interface WidgetHeader`.
- **#8950** deletes `android-widget.model.ts`, `android/widget-data.service.ts`
  and `android-widget.effects.ts`; renames `android-widget.selectors.ts` →
  `src/app/features/widget/store/widget.selectors.ts`; and modifies the same
  `WidgetData.kt` / `WidgetDataTest.kt`.

**The trap.** #8950's `widget-data.model.ts` carries `validUntil?` (written by
`serializeWidgetData` in `widget-data.service.ts`) but has **no `dayStr`**. On
rebase the TS conflicts as modify/delete, and the obvious resolution — *"the file
moved, take mine"* — drops the stamp from the writer while the Kotlin reader
(taken from #9118) keeps asking for it. Missing stamp makes the reader fall back
to trusting the list, i.e. labelling a stale list **"Today"**.

That is exactly the #9098 bug, resurrected, **with every test green**: the Kotlin
tests parse hand-written JSON and never see the TypeScript writer, and the
TypeScript tests never see Kotlin. Nothing in CI connects the two ends.

**Required work, not just conflict resolution:**

1. Port `dayStr` into `src/app/features/widget/widget-data.model.ts` and emit it
   from `widget.selectors.ts` / `widget-data.service.ts`. `validUntil` alone is
   not enough — the reader needs both, and #9118 deliberately keeps them
   independent (an untrustworthy `validUntil` must never pair with a `dayStr`).
2. Keep #9118's Kotlin reader semantics through the `WidgetData.kt` merge.
3. Decide the iOS side: `WidgetData.swift` reads `validUntil` but has no `dayStr`
   equivalent, so it has no "Outdated (date)" header — only expired/not-expired.
   Either port the header or consciously accept the gap and write it down.
4. **Add a test that fails if the writer stops emitting a field the readers
   require.** The whole failure mode is that no existing test spans the writer
   and either reader. Without this, the same trap is re-armed for the next person.

## 3. Apple provisioning

Blocks every signed build. Pure portal work; nothing in the repo needs changing —
identifiers were verified consistent (App Group `group.com.super-productivity.app`
matches across `App.entitlements`, `SupWidget.entitlements` and
`WidgetShared.swift`; widget bundle ID `com.super-productivity.app.widget` matches
the assertion in `build-ios.yml`).

Steps (also in [`ios/App/SupWidget/README.md`](../../ios/App/SupWidget/README.md)):

1. Portal: register App ID `com.super-productivity.app.widget`.
2. Portal: create App Group `group.com.super-productivity.app`, enable on **both**
   App IDs.
3. Portal: regenerate the **app's** distribution profile (must now carry the App
   Group) **and** create a distribution profile for the widget App ID.
4. CI: set **both** secrets.

```bash
base64 -w0 app.mobileprovision    | gh secret set IOS_PROVISION_PROFILE
base64 -w0 widget.mobileprovision | gh secret set IOS_WIDGET_PROVISION_PROFILE
```

> **Two secrets change, not one.** Adding only `IOS_WIDGET_PROVISION_PROFILE`
> satisfies the `exit 1` guard in `build-ios.yml` and the build goes green — but
> the app profile then lacks the App Group, `UserDefaults(suiteName:)` returns
> nil, the bridge rejects every write, and the widget renders empty forever with
> nothing naming the cause.

**Verify before merge, not at release.** `build-ios.yml` runs only on `v*` tags,
so no PR check ever exercises signing. Use its `workflow_dispatch` against the
branch to prove the path works without cutting a release.

## 4. The widget itself

Still broken; symptom not characterised. What today's green CI narrows:

- The Xcode project builds and the SupWidget target compiles.
- `SupWidgetTests` passes — the Swift parser matches the golden JSON shared with
  Kotlin and the TS writer.

What it does **not** cover: `ios-pr.yml` builds with `CODE_SIGNING_ALLOWED=NO`, so
it never touches signing, provisioning, or the App Group container — precisely
where a "renders empty forever" failure lives. Consistent with both a
provisioning artifact and a runtime bridge bug; rules out neither.

Note §1 is a candidate cause in its own right: if the reader wants a stamp the
writer never sends, the widget misbehaves without any code looking wrong.

**Iteration cost:** the maintainer is on Linux and cannot build iOS locally. Every
attempt is a macOS CI round-trip. Factor this into whether to finish or park.

## 5. Verification gaps

- **Android device smoke test** — non-negotiable before merge. The Android widget
  is a working shipping feature that this PR rewrites: the drain gate moved from
  `isAllDataLoadedInitially$` to `afterInitialSyncDoneStrict$`, and the new
  fail-closed guard's failure path is a silent `Log.err` + `return` with no
  snackbar. Unit tests pass; nobody has tapped a checkbox on a real home screen.
  Also re-check #9098's own case (clock past the boundary → header flips).
- **`agvtool` version sync** — CI sets versions via `new-marketing-version` +
  `new-version -all`. Both targets define the settings so it *should* update them
  in lockstep, and it must: the App Store rejects an extension whose version
  differs from its containing app. Untested from Linux; watch on the first tagged
  build.
- **Deferred-write gap (accepted, deliberate)** — the dropped
  `recordDeferredWriteFailure` pair has no master equivalent. Master's flag is set
  on permanent/invalid deferred failures but not on transient retry-exhaustion, so
  the widget can drain and ack a lease while transiently-failed deferred actions
  sit unpersisted. Narrow (the buffer's loss-on-reload is a pre-existing op-log
  issue, not widget-caused), but real. If the widget needs a recoverable variant,
  that is its own PR against the sync model — not a ride-along here.

## 6. Known costs this PR locks in

Not blockers; decide consciously rather than by default.

- `npm test` now also runs `generate:ios-widget-localizations:test`, so an iOS
  tooling break fails the whole suite for everyone, including web-only
  contributors.
- `sync:ios` depends on the generator.
- The 18 `WIDGET.IOS.*` keys carry Swift `%@` specifiers into `en.json`, so they go
  to translators for all 28 locales and ship in every user's bundle — while the
  generator only emits `en.lproj`, leaving the widget English-only regardless.
- `ios-pr.yml` puts a macOS runner on every PR touching `ios/**` or
  `src/app/features/widget/**`.
- `IOS_CAPABILITIES.homeWidget: true` is currently inert — `homeWidget` is read
  nowhere in production code.
